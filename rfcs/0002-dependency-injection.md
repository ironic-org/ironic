# RFC 0002: Dependency Injection

- Status: Accepted for initial implementation
- Target: Ironic 0.1
- Authors: Ironic contributors

## Summary

Ironic dependency injection uses type-based provider keys, asynchronous type-erased factories, singleton and transient scopes, module-aware visibility, and immutable override layers created before compilation. All provider construction uses one asynchronous internal contract; synchronous constructors are adapted into immediately ready futures.

## Motivation

Rust applications need explicit ownership and type safety, while real providers often require asynchronous setup. A single construction model avoids maintaining separate resolution engines and allows a synchronous `Injectable` implementation to coexist with async factories.

## Goals

- Resolve concrete Rust types without string tokens.
- Support singleton, transient, value, and async factory providers.
- Make dependency graphs and failures observable.
- Prevent duplicate singleton construction under concurrency.
- Support test overrides without global mutable state.
- Respect module ownership and visibility from RFC 0001.

## Non-goals

- Request-scoped providers.
- Property or field mutation injection.
- Runtime reflection.
- Automatic construction of arbitrary unregistered types.
- Automatic trait-object binding inference.
- Circular dependencies through lazy proxy objects.

## Provider keys

```rust
use std::any::{TypeId, type_name};

#[derive(Clone, Copy, Debug, Eq, Hash, PartialEq)]
pub struct ProviderKey {
    type_id: TypeId,
    type_name: &'static str,
}

impl ProviderKey {
    pub fn of<T: Send + Sync + 'static>() -> Self {
        Self {
            type_id: TypeId::of::<T>(),
            type_name: type_name::<T>(),
        }
    }
}
```

Equality uses `TypeId`; `type_name` is diagnostic metadata. The initial public resolution API returns `Arc<T>` because the container owns provider lifetime and singleton sharing is common.

```rust
let users = container.resolve::<UsersService>().await?;
```

## Scope

```rust
#[derive(Clone, Copy, Debug, Eq, PartialEq)]
pub enum Scope {
    Singleton,
    Transient,
}
```

- A singleton is constructed at most once per compiled application container. Every successful resolution returns an `Arc` pointing to the same value.
- A transient is constructed on every resolution. Dependencies may still be shared singleton values.
- Singleton is the default scope.
- Request scope is deferred because it requires a separate child-container and cancellation design.

## Construction contract

Provider construction is asynchronous internally:

```rust
use std::{any::Any, future::Future, pin::Pin, sync::Arc};

type ErasedProvider = Arc<dyn Any + Send + Sync>;
type ProviderFuture<'a> = Pin<
    Box<dyn Future<Output = Result<ErasedProvider, ResolveError>> + Send + 'a>
>;

pub trait ProviderFactory: Send + Sync + 'static {
    fn create<'a>(&'a self, resolver: Resolver<'a>) -> ProviderFuture<'a>;
}
```

`Resolver` is a limited view of the container scoped to the requesting module and current resolution path. Factories cannot register or override providers while resolving.

Synchronous constructors implement a user-facing trait:

```rust
pub trait Injectable: Send + Sync + Sized + 'static {
    fn inject(dependencies: ResolvedDependencies) -> Result<Self, ResolveError>;
}
```

Their provider definition wraps `inject` in the asynchronous factory contract. An explicit async factory is registered with an async closure or a `ProviderFactory` implementation.

This decision means there are not separate synchronous and asynchronous containers. All resolution methods are async, including resolution of synchronous providers.

## Definitions and dependency metadata

```rust
pub struct ProviderDefinition {
    pub key: ProviderKey,
    pub scope: Scope,
    pub dependencies: Vec<Dependency>,
    pub factory: Arc<dyn ProviderFactory>,
    pub eager: bool,
}

pub struct Dependency {
    pub key: ProviderKey,
    pub optional: bool,
}
```

Dependencies are declared metadata, generated by macros or supplied through explicit builders. This supports graph validation before execution. Factory code may perform dynamic resolution, but runtime cycle and visibility checks still apply.

The initial builder shapes are:

```rust
ProviderDefinition::singleton::<UsersService>()
    .dependency::<UserRepository>()
    .constructor(|deps| {
        Ok(UsersService::new(deps.require::<UserRepository>()?))
    });

ProviderDefinition::factory::<DatabasePool>(Scope::Singleton)
    .async_constructor(|resolver| async move {
        let config = resolver.resolve::<DatabaseConfig>().await?;
        DatabasePool::connect(&config.url).await.map_err(Into::into)
    });
```

Exact builder naming may change during implementation, but the metadata and construction semantics are normative.

## Trait-object dependencies

The 0.1 strategy is an explicit newtype token. Ironic will not initially expose `bind::<dyn Trait>().to::<Implementation>()`.

```rust
pub trait UserRepository: Send + Sync {
    fn find(&self, id: UserId) -> RepositoryFuture;
}

pub struct UserRepositoryToken(pub Arc<dyn UserRepository>);
```

`UserRepositoryToken` is an ordinary concrete provider key. Applications register a value or factory for it and inject `Arc<UserRepositoryToken>`.

This approach is verbose but fully type checked, supports multiple semantic bindings through distinct newtypes, and avoids unstable pointer metadata or surprising `TypeId` behavior. A future ergonomic binding API may generate equivalent token providers.

## Registration

Registrations are produced by the module compiler. A container cannot be mutated after successful compilation.

- Duplicate keys in one module are rejected.
- The same key in different modules represents distinct registrations.
- The requesting module selects a local registration first, then one unambiguous exported registration from a direct import.
- A provider factory resolves dependencies with the owning module as its visibility context.

Controllers use the same resolution engine but are not exportable provider registrations in 0.1.

## Singleton state machine

Each singleton registration owns an async once cell with these conceptual states:

```text
Uninitialized ──resolve──▶ Initializing ──success──▶ Ready
      ▲                         │
      └──────── retry ◀── failure
```

- Concurrent callers await the same initialization attempt.
- A successful value is published once.
- A failed construction is not cached permanently; a later independent resolution may retry.
- Every waiter for the failed attempt receives the same structured failure context.
- Runtime resolution-path tracking detects re-entry into a singleton already being initialized by the same logical resolution chain and reports a cycle instead of deadlocking.

The initial implementation should use Tokio's async synchronization primitives because Tokio is the framework runtime. The public DI abstractions must not expose a concrete once-cell type.

## Resolution paths and cycle detection

Every top-level resolution creates an immutable logical path extended at each dependency edge:

```text
UsersController -> UsersService -> UserRepository -> DatabasePool
```

If the next provider registration already appears in the current path, resolution returns `CircularDependency`. Identity includes the owning module and provider key so same-key providers in different modules remain distinguishable.

Static dependency metadata is checked during module compilation. Runtime path checking covers async factories and future conditional resolution.

## Optional dependencies

Optional dependencies must be declared explicitly. Missing optional dependencies resolve to `None`; construction failures for a registered optional dependency remain errors.

```rust
let cache = deps.optional::<CacheClient>()?;
```

The general `resolve::<T>()` API never silently converts a missing provider to `None`.

## Value and alias providers

Value providers store a caller-supplied `Arc<T>` and always behave as singletons.

Alias providers are deferred from the 0.1 kernel. Wrapper-token value/factory providers cover the primary trait-binding use case without creating ambiguous ownership and downcast rules.

## Overrides

Overrides are immutable input to test compilation:

```rust
let test_module = TestModule::builder::<UsersModule>()
    .override_value(MockClock::fixed(now))
    .override_factory::<UserRepositoryToken>(/* ... */)
    .compile()
    .await?;
```

Rules:

- Overrides are keyed by the owning module plus provider key.
- A convenience override by provider key succeeds only when exactly one visible registration matches.
- Overrides are applied to definitions before graph validation and container construction.
- Overrides never mutate a running application or a process-global registry.
- Scope may be replaced explicitly; otherwise the original scope is preserved.

## Error behavior

`ResolveError` is structured and preserves a resolution path. Initial error codes:

- `RF_DI_MISSING_PROVIDER`
- `RF_DI_PRIVATE_PROVIDER`
- `RF_DI_AMBIGUOUS_PROVIDER`
- `RF_DI_DUPLICATE_PROVIDER`
- `RF_DI_CIRCULAR_DEPENDENCY`
- `RF_DI_FACTORY_FAILED`
- `RF_DI_TYPE_MISMATCH`
- `RF_DI_INVALID_OVERRIDE`

Factory errors may attach a source error for logging, but public display must avoid leaking secrets. Type mismatches indicate a framework or unsafe manual-definition bug and should include expected and actual diagnostic type names when available.

## Concurrency and cancellation

- Provider values must be `Send + Sync + 'static`.
- Factory futures must be `Send`.
- Cancelling one waiter must not cancel shared singleton initialization while other waiters exist.
- Application shutdown may cancel unfinished eager initialization as part of application-build cancellation.
- The container holds strong references to singleton values until shutdown completes.

The exact shared-initialization task mechanism is an implementation detail and requires a dedicated cancellation regression test.

## Alternatives considered

### Synchronous provider trait only

Rejected because databases, clients, and other resources commonly need async initialization, and blocking inside constructors is unsafe on async runtimes.

### Separate sync and async resolution engines

Rejected because dependency chains can mix both kinds, producing duplicate APIs and complicated macro behavior.

### Service locator with runtime registration

Rejected because it weakens graph validation and makes application state depend on execution order.

### Automatic trait-object bindings

Deferred because explicit newtype tokens are clearer and support multiple bindings without extra qualifiers.

### Caching singleton failures permanently

Rejected because transient infrastructure failures could permanently poison an otherwise recoverable container. Retry behavior will be documented and observable.

## Performance impact

Resolution performs a module-scoped registration lookup and an async once-cell access for singletons. Transients additionally allocate and run their factory. Declared dependencies may be indexed during compilation. Benchmarks will measure warm singleton resolution, transient resolution, concurrent first resolution, and error-path construction.

## Testing strategy

- Unit tests for keys, definitions, values, and downcasting.
- Singleton identity and exactly-once concurrency tests.
- Transient distinct-instance tests.
- Mixed sync/async dependency-chain tests.
- Static and runtime circular-dependency tests.
- Failed singleton attempt and retry tests.
- Cancellation tests for concurrent singleton waiters.
- Module visibility and same-key ownership tests.
- Override isolation tests proving two test containers cannot affect each other.
- Error snapshot tests with secret redaction.

## Migration considerations

New provider scopes can extend `Scope` non-exhaustively before 1.0. A future trait-binding API should compile into ordinary token providers so the initial wrapper-token model remains valid.

## Unresolved questions

The implementation spike must choose an async once-cell/shared-task mechanism that satisfies the cancellation rules. This is an internal choice and does not block the public contract.
