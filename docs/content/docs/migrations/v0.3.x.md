# Migration Guide — v0.3.x

This guide covers breaking changes introduced in the v0.3.x release line.

---

## 1. RateLimitMiddleware now requires a backend

**Old API (v0.2.x):**

```rust
RateLimitMiddleware::new(100, 60)  // max_requests per window_secs
```

**New API (v0.3.x):**

```rust
use std::sync::Arc;
use ironic::security::rate_limit::{InMemoryRateLimiter, RateLimitMiddleware};

let backend = Arc::new(InMemoryRateLimiter::new());
RateLimitMiddleware::new(backend, 100, 60)
```

### Why?

The `RateLimitBackend` trait was extracted so you can swap the storage without
changing middleware logic.  `InMemoryRateLimiter` preserves the old in-process
behaviour.  For multi-replica deployments, use `RedisRateLimiter`.

### Migration steps

1. Add `use std::sync::Arc;`
2. Add `use ironic::security::rate_limit::InMemoryRateLimiter;`
3. Create a backend: `let backend = Arc::new(InMemoryRateLimiter::new());`
4. Pass it as the first argument: `RateLimitMiddleware::new(backend, 100, 60)`

## 2. Metrics API renamed from MetricsStore to MetricsRegistry

**Old API (v0.2.x):**

```rust
use ironic::metrics::MetricsStore;

fn record(metrics: Inject<MetricsStore>) {
    metrics.counter("orders_total").with_label("status", "confirmed").inc();
}
```

**New API (v0.3.x):**

```rust
use ironic::metrics::MetricsRegistry;

fn record(metrics: Inject<MetricsRegistry>) {
    let counter = metrics.counter("orders_total");
    counter.inc();
}
```

### What changed

| Old | New | Reason |
|-----|-----|--------|
| `MetricsStore` | `MetricsRegistry` | Clearer naming: "registry" implies shared, central metric store |
| `counter(name).with_label(k, v).inc()` | `metrics.counter(name).inc()` | Simplified API — labels are handled by the registry |
| `Mutex`-guarded counters | `AtomicU64` counters | Lock-free for high-throughput paths |

### Migration steps

1. Replace `MetricsStore` with `MetricsRegistry` in imports and type annotations
2. Replace `metrics.counter("x").with_label("k", "v").inc()` with `metrics.counter("x").inc()`
3. Replace `metrics.gauge("x").with_label("k", "v").set(n)` with `metrics.gauge("x").set(n)`
4. Replace `MetricsModule` imports — the module name is unchanged

## 3. Session middleware API change

**Old API (v0.2.x):**

The session middleware was tightly coupled to cookie-based sessions with no
pluggable backend.

**New API (v0.3.x):**

```rust
use ironic::auth::sessions::{SessionStore, InMemorySessionStore, RedisSessionStore};

// In-memory (development)
let store = Arc::new(InMemorySessionStore::new());

// Redis (production)
let store = RedisSessionStore::new(connection_manager)
    .with_ttl(Duration::from_secs(86400));
```

### Why?

The `SessionStore` trait was introduced to support multiple backends.  The
old cookie-based API was removed in favour of explicit store configuration.

### Migration steps

1. Create a store instance (`InMemorySessionStore` or `RedisSessionStore`)
2. Wrap it in `Arc` and inject into your auth module
3. Replace direct cookie manipulation with `session.insert()` / `session.get()`
4. Call `store.save(&session)` after mutations

## 4. Health endpoint returns composite format

**Old API (v0.2.x):**

```json
{"status": "ok"}
```

**New API (v0.3.x):**

```json
{
  "status": "ok",
  "checks": {
    "database": "ok",
    "redis": "ok"
  }
}
```

### What changed

The `/health` endpoint now returns per-component check results alongside the
aggregated status.  If you had monitoring rules parsing the old single-field
response, update them to read `checks.{name}`.

### HTTP status codes

| Indicator states | HTTP status |
|------------------|-------------|
| All ok | `200 OK` |
| Any degraded | `207 Multi-Status` (was 200) |
| Any unhealthy | `503 Service Unavailable` (unchanged) |

### Migration steps

1. Update monitoring / alerting rules to expect the new JSON format
2. If you relied on the 200 status for degraded states, switch to checking
   `207 Multi-Status`
3. Review your health check scripts: the `/health` endpoint now supports
   the `HealthIndicator` trait for custom checks

## 5. ConfigurationLoader::auto_detect_env behaviour

**What changed:**

`auto_detect_env()` now checks `IRONIC_ENV` before `APP_ENV` with a fallback
to `"development"`.  Previously the fallback was empty, which skipped profile
loading silently.

**If you relied on `profile()` being a no-op when neither env var was set:**
Your app will now attempt to load `config.development.toml`.  If the file
doesn't exist it is silently skipped — identical behaviour to before.

**Migration steps:**

- If you have a `config.development.toml`, verify its values override the
  base config correctly
- If you don't want any profile loading, remove the `auto_detect_env()` call

## 6. Static file serving now requires `static-files` feature

**What changed:**

Static file serving was moved behind the `static-files` feature flag.

**Old Cargo.toml:**

```toml
ironic = { features = ["full"] }
```

**New Cargo.toml:**

```toml
ironic = { features = ["static-files"] }
```

### Migration steps

1. Add `"static-files"` to your features list in `Cargo.toml`
2. If you use `StaticFileConfig`, it is now exported from the crate root

---

## 7. Multipart upload feature flag

**What changed:**

Multipart upload support is behind the `multipart` feature flag.

**New Cargo.toml:**

```toml
ironic = { features = ["multipart"] }
```

### Migration steps

1. Add `"multipart"` to your features list in `Cargo.toml`
2. Import types from `ironic::MultipartForm`, `ironic::UploadedFile`, etc.

---

## 8. ConcurrencyLimitLayer requires `resilience-ext` feature

**New Cargo.toml:**

```toml
ironic = { features = ["resilience-ext"] }
```

This enables `AxumAdapter::max_concurrent_requests(n)` and the
`ConcurrencyLimitLayer` / `ConcurrencyLimitService` types.

---

## 9. Backtrace feature

Error backtraces are gated behind the `backtrace` feature:

```toml
ironic = { features = ["backtrace"] }
```

When enabled, `HttpError::internal()` automatically captures a
`Backtrace`.  Backtraces are serialized in debug builds only.
